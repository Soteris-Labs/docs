Use this checklist before adding new docs pages, fields, examples, or agent instructions.

## Line tests

| Test | Public-safe version |
| --- | --- |
| Interface, not judgment | Explain how systems interoperate, not how Soteris decides. |
| State, not criteria | Name lifecycle states without publishing review criteria. |
| Verify-side, not issue-side | Show what a receipt proves, not how proof is constructed internally. |
| Curated subset, not backend dump | Publish stable external shapes, not raw internal schemas. |
| Shape, not substance | Use synthetic examples and avoid real client data. |

## Red flags

Do not publish fields or pages that expose confidence scores, gate rationale, vote detail, prompt text, internal procedure identifiers, scoring utilities, raw evidence registers, or implementation-specific hash construction.

## Default answer

If a page helps a partner or agent safely interoperate with Soteris, it is likely in scope.

If a page explains how Soteris reaches professional judgment, it is out of scope.
