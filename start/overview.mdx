## Why Soteris

Assurance workflows need more than generated documents. They need durable state that everyone, including software, can trust.

Traditional assurance work is coordinated through documents, inboxes, shared drives, spreadsheets, and manual status pings. Those tools hold evidence, but they do not create a shared operational source of truth. It stays unclear what is official, who can act, what is blocked, what has been reviewed, and what has actually been released.

Soteris makes that progression explicit. Lifecycle state says where an engagement stands. Controlled inputs tie submissions to the engagement. Review authority keeps professional approval separate from automated output. Released artifacts carry their own access boundaries. Verification records let permitted parties confirm what was released without exposing how it was concluded.

## From static workpapers to operational assurance

A workpaper is a snapshot. It records what someone concluded at one moment, in a format built for a human to read later. It cannot tell you whether it is current, whether it was superseded, who is allowed to act on it next, or whether anything downstream is blocked.

The operational gap is not the document. It is everything around the document: the state, the ownership, the sequence, and the proof.

Soteris closes that gap by treating the engagement, not the file, as the unit of record. Every input, artifact, and release is bound to a lifecycle state and an actor boundary. Agents can accelerate preparation against that structure. The portal remains the authority for what is true right now.

## How the system fits together

Soteris is organized around five layers.

1. **Portal.** The system of record for engagement state, access, inputs, artifacts, and released deliverables.
2. **Backend control boundary.** The command and persistence layer that validates transitions, records events, and stores references.
3. **Agent runtime.** The private execution layer that prepares findings, drafts, classifications, and recommendations.
4. **Review and release.** The governed layer where Soteris controls approval, release readiness, and final authorization.
5. **Verification.** The inspection layer for released records and receipts.

Together these layers make assurance work legible: what exists, who can act, what changed, what was released, and what can be confirmed.

## Using these docs with an AI agent

These docs are written to be read by agents as well as people. An agent can safely map the lifecycle, resolve field names, inspect record shapes, and plan a read path against the portal. It should not treat any surface here as a live write endpoint, and it should stop before attempting to mutate engagement state.

Start an agent at the [Agent guide](/start/agent-guide), which lists safe tasks, stop conditions, and a recommended discovery path.
